#### 步骤 3: 解析 Action (232-253行)


# 3.1 提取 Action
action_match = re.search(r"Action: (.*)", llm_output, re.DOTALL)
action_str = action_match.group(1).strip()

# 3.2 检查是否结束
if action_str.startswith("Finish"):
    final_answer = re.match(r"Finish\[(.*)\]", action_str).group(1)
    break

# 3.3 解析工具调用
tool_name = re.search(r"(\w+)\(", action_str).group(1)
args_str = re.search(r"\((.*)\)", action_str).group(1)
kwargs = dict(re.findall(r'(\w+)="([^"]*)"', args_str))

## 🔄 ReAct 框架核心思想


┌─────────────────────────────────────────┐
│  ReAct = Reasoning (推理) + Acting (行动) │
└─────────────────────────────────────────┘

循环流程:
1. Thought (推理): LLM 思考下一步该做什么
2. Action (行动): 执行具体的工具调用
3. Observation (观察): 记录工具返回的结果
4. 重复 1-3，直到任务完成


### ReAct vs 传统方法

| 传统方法 | ReAct 方法 |
|---------|-----------|
| 一次性生成答案 | 多轮推理和行动 |
| 无法使用外部工具 | 可以调用工具 |
| 信息有限 | 可以查询实时信息 |
| 不可控 | 每一步可观察 |

## ⚠️ 注意事项

### 1. 循环次数限制

for i in range(5):  # 最多 5 轮

**原因**: 防止 LLM 陷入无限循环

### 2. API 密钥安全

# ❌ 不要硬编码在代码中
API_KEY = "sk-xxxxx"

# ✅ 应该从环境变量读取
API_KEY = os.getenv("API_KEY")


### 3. 错误处理
- 每个工具函数都有完善的 try-except
- LLM 调用也有错误处理
- 解析失败时会给出提示并继续循环

### 4. LLM 输出格式
- 依赖 LLM 严格遵循 `Thought: ... Action: ...` 格式
- 如果 LLM 不遵循格式，会返回错误提示

## 📈 性能优化建议

1. **缓存天气结果**: 避免短时间内重复查询
2. **流式输出**: 使用 `stream=True` 提升用户体验
3. **并行工具调用**: 如果工具之间无依赖，可以并行执行
4. **Token 优化**: 定期清理历史记录，保留关键信息
